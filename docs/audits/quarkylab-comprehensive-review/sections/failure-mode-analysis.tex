\section{Failure-Mode and Dependency Analysis}\label{sec:fma}

This section walks the credible failure scenarios for the estate. It is analytical only:
\textbf{no failure injection was performed}, consistent with the read-only mandate. The
dependency structure it draws on is in \code{diagrams/service-dependencies.mmd} and the
recovery order in \code{diagrams/recovery-dependencies.mmd}.

The single most important structural observation is a small set of keystone dependencies:
OPNsense (routing/DHCP/DNS egress), Randy (all backups, NFS, and k8s storage), the single
Unbound resolver, and the co-located observability stack. Most high-impact scenarios below
trace back to one of these four.

\begin{callout}
Several of these are safely testable in production with a maintenance window (fail one
Pi-hole, unplug WAN1 to exercise 5G failover, drain a node, restore a guest to a scratch
VMID, test-decrypt break-glass). The genuinely unsafe tests - dual-node loss, storage
loss, real power outage, battery failure - should only be modeled, never injected.
\end{callout}
