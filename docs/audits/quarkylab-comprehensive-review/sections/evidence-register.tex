\section{Evidence Register}\label{sec:evidence}

Every finding in this review traces to a numbered evidence item. The register below
records the source, the system it came from, the exact command or path, its age, a
reliability tier, and the limitations that bound how far it can be trusted. The
machine-readable form is \code{data/evidence-register.csv}.

\subsection{Reliability tiers and evidence types}

Evidence is typed as \emph{live} (a direct read-only query executed during this
review, 2026-07-23), \emph{live-captured} (the prior audit's node captures, one day
old), \emph{generated} (assessment reports produced by prior tooling),
\emph{inferred} (conclusions read out of source code or configuration), or
\emph{manual} (operator-maintained documentation). Live evidence is weighted
highest for current-state claims; documented evidence is weighted highest for
intent and history. Where the two conflict, the conflict itself is recorded rather
than silently resolved: the most consequential examples are the GPU assignment
(live overrides the specification, DR-01), the OPNsense version (documented as
25.1.12, not re-verified live, E-23 / DR-02), and Randy's PVE version, where the
live reading of 9.2.3 supersedes the documented 9.1.1 (E-04 / DR-10).

\subsection{Notable conflicts surfaced}

The register makes three conflicts explicit. First, the review specification's GPU
"correction" is inverted versus live hardware and must not be applied (E-02 / E-03).
Second, the OPNsense version is documented as 25.1.12 but was not re-verified live;
roughly a dozen references still say 25.7, which is the future CARP target, not the
running version (E-23). Third, Randy reports PVE 9.2.3 live while the documentation
still says 9.1.1, a drift the prior audit did not catch (E-04, DR-10).
