\section{Monitoring, Logging, and Alerting}\label{sec:monitoring}

\subsection{Stack and topology}

Observability is a Grafana, Prometheus, Loki, InfluxDB, and Scrutiny stack running together in
CT103 on pve4 (\code{192.168.10.183}), live-confirmed (E-06). Alerting uses Grafana Unified
Alerting straight to Discord with no Alertmanager. Two contact points exist: an infrastructure
channel (\code{discord-alerts}) and a UPS channel (\code{discord-ups}). Prometheus was reported
at 15 of 15 targets up (E-15), which means the silent not-scraping failure mode is clean.
Config lives as code in a private repository, which is a genuine operational-maturity strength.

\subsection{Blind spots and self-dependency}

\begin{itemize}
\item \rh{Single alert egress (QL-R09, DETECT-01d).} All alerts leave over one Discord webhook
across the WAN. A WAN or Discord outage blinds alerting, and the deterministic node-down message
shares that path. A second out-of-band channel is recommended (QL-REC14).
\item \rh{Co-located stack (QL-R10).} Metrics, logs, and alert evaluation all run in one CT on
one node. Losing that node loses monitoring and stops alert evaluation.
\item \rh{Loki unauthenticated.} \code{auth\_enabled: false} on VLAN 1 makes logs readable and
poisonable by any VLAN 1 host.
\item \rh{SIEM scope (QL-R22).} Wazuh agents run only on the eight hypervisors; guests, OPNsense,
the switch, and endpoints are blind to the SIEM.
\end{itemize}

\begin{callout}[title=The silent-alert lesson]
The degraded \code{bulk} pool was masked for roughly 71 hours. The \code{backup-verify} job ran
as an Ares user cron; the laptop suspends overnight with no catch-up, so three nightly runs were
silently dropped and \code{backup\_verify\_*} froze. The condition was ultimately surfaced by
\code{BackupVerifyReportStale}, not by \code{ZfsPoolDegraded}. Whether \code{ZfsPoolDegraded}
actually fired to Discord is unverified read-only (QL-R11, R-02) \sevhigh. A silent critical is
worse than no monitoring; verifying that delivery path is the single highest-value monitoring
action. The scheduler itself has since been moved to a persistent systemd user timer.
\end{callout}

\subsection{What is done well}

Dead-man heartbeats already exist (\code{NetframeMonitorStale}, \code{NetworkSyslogSilent}, both
alerting on noData), a corosync quorum textfile alert has been added, WAN-down detection is wired
from OPNsense dpinger, and the entire configuration is version-controlled. The recommended next
steps are a read-only \code{prometheus-pve-exporter} for guest and quorum metrics (QL-REC06), the
backup-stale, verify-failed, and broad cert-expiry rules (QL-REC07), and the second alert channel
(QL-REC14).
