## Why private repos use pre-push hooks, not branch protection
On the **free plan**, both branch protection *and* rulesets are **Pro-gated for private repos** (HTTP 403). Options were: pay for Pro, make public (no - they reveal topology), or a **client-side pre-push hook**. Chose the hook.

- Each private repo has `.githooks/pre-push` (runs the same checks as its CI) + `install-hooks.sh`.
- **Per clone:** `./install-hooks.sh` (sets `core.hooksPath → .githooks`).
- **Bypass** (sparingly): `git push --no-verify`.
- **Tools** (installed on Ares `~/.local/bin`, on PATH): `ruff`, `shellcheck`, `yamllint`. If a tool is missing the hook *skips* that check (warns) rather than blocking.
- CI still runs server-side on the private repos too - advisory (visible red/green), just not a hard merge gate.

## Deployment policy (the "CD" that isn't)
No GitHub Actions deploy to the cluster - that would need a LAN-connected self-hosted runner and hand a cloud CI a path into the infrastructure. **Deploy remains manual**: SSH/systemd/API (e.g. the pt-exec + provisioning flows used for the monitoring stack and Pi-hole). CI is the **quality gate before** a human deploys, not the deployer.

## To add CI to a new repo
Copy the matching `.github/workflows/ci.yml` (+ `.yamllint` if YAML). For a private repo also add `.githooks/pre-push` + `install-hooks.sh` and run the installer.
