## Why private repos use pre-push hooks, not branch protection
On the **free plan**, both branch protection *and* rulesets are **Pro-gated for private repos** (HTTP 403). Options were: pay for Pro, make public (no - they reveal topology), or a **client-side pre-push hook**. Chose the hook.

- Each private repo has `.githooks/pre-push` (runs the same checks as its CI) + `install-hooks.sh`.
- **Per clone:** `./install-hooks.sh` (sets `core.hooksPath → .githooks`).
- **Bypass** (sparingly): `git push --no-verify`.
- **Tools** (installed on Ares `~/.local/bin`, on PATH): `ruff`, `shellcheck`, `yamllint`. If a tool is missing the hook *skips* that check (warns) rather than blocking.
- CI still runs server-side on the private repos too - advisory (visible red/green), just not a hard merge gate.

## Deployment policy (the "CD" that isn't)
No GitHub Actions deploy to the cluster - that would need a LAN-connected self-hosted runner and hand a cloud CI a path into the infrastructure. **Deploy remains manual**: SSH/systemd/API (e.g. the pt-exec + provisioning flows used for the monitoring stack and Pi-hole). CI is the **quality gate before** a human deploys, not the deployer.

## Do NOT path-filter a workflow that provides a required check (fixed 2026-07-15)

**Symptom:** a docs/vault/runbook-only PR to Home-Lab sat at `mergeStateStatus: BLOCKED` with `"checks": 0` and could only be merged with `gh pr merge N --squash --admin`. PRs #9 and #10 both went in that way.

**Cause:** `ci.yml` supplies **both** required status checks on `main` (`Python lint + syntax`, `ShellCheck (bash)`), but it was path-filtered to `scripts/**` + `.github/workflows/**`. A docs-only change matches neither, so the workflow never ran, and **GitHub treats a required check that never ran as *pending*, not *passed*** - blocked forever. Nothing was broken; the gate was simply unsatisfiable.

**Fix (PR #11):** dropped the path filter from `ci.yml` so it runs on every push and PR. Costs a few seconds against an unchanged `scripts/`; in exchange the gate means what it says and `--admin` stops being routine. Verified by merging a docs-only PR with no bypass.

`diagram.yml` (`topology/**`) and `netlab.yml` (`netlab/**`) **keep** their filters - they are expensive (LaTeX / containerlab) and are **not** required checks, so skipping them is harmless.

> [!warning] The rule
> Path-filter freely, but **never on a workflow whose job name is a required status check**. If a required job must be skippable, it needs a companion no-op workflow reporting the same context name on the inverse paths - which means keeping two path lists in sync forever. Simpler to just let it run.

Home-Lab's default branch is **`main`** (not `master`), `enforce_admins=false` (owner can direct-push), `strict=false`, and required reviews are off.

## To add CI to a new repo
Copy the matching `.github/workflows/ci.yml` (+ `.yamllint` if YAML). For a private repo also add `.githooks/pre-push` + `install-hooks.sh` and run the installer. If the repo has branch protection, make sure the required-check contexts come from a workflow that is **not** path-filtered (see above).
